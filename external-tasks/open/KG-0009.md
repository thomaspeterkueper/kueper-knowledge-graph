## Open questions

**1. PHY-1101 and CHE-1101 do not exist in any current export.** If `MAT-1101.module_unlocks` references them now, that recreates the exact dangling-reference pattern this whole thread started with (the original missing `MAT-1001`). Two options: (a) define minimal stub records now - id, title, subject, `status: planned`, `requires: [LRN:SSF:MAT-1101]`, empty assets, matching the pattern already used for `AST-1101`/`EAR-2201` etc. - or (b) leave `MAT-1101.module_unlocks` pointing at them as accepted forward references until dedicated PHY/CHE tickets land, and track that explicitly rather than silently. SSF has no preference; (a) avoids a repeat of the earlier problem, (b) keeps this request scoped to Mathematics only.

**2. AST-1101 already exists with `dependencies.requires: []`.** Adopting this tree means editing that existing canonical record to `requires: [LRN:SSF:MAT-1101]` - a modification to something already published, not just new creation. Flagging explicitly since it's a different kind of change than the rest of this request.

**3. MAT-2101/2102/2103 vs the already-existing MAT-2201/2202.** `MAT-2102` "Bayes" and the existing `LRN:SSF:MAT-2201` ("Bayes-Inferenz fuer Wissenschaftstexte") appear to be the same concept; likewise `MAT-2103` "Differentialgleichungen" and `LRN:SSF:MAT-2202` ("Differentialgleichungen als Modellsprache"). `MAT-2201` already carries real `archive_unlocks`/`related`/`noxia.grants` data. Should `MAT-2102`/`MAT-2103` be dropped in favour of renumbering `MAT-2201`->`MAT-2102` and `MAT-2202`->`MAT-2103` into this tree (preserving their existing data), rather than creating near-duplicate records?
